## 0. 범위와 비범위

### 이 스펙이 다루는 것
- **기능 1: 연구 질문 캔버스** — 연구 아이디어를 구조화 입력하고, 정합성 경고를 주며, 1페이지 컨셉 노트 초안을 생성한다.
- **기능 2: 연구 유형 분기형 마법사** — 6개 연구 유형 중 하나를 고르면 체크리스트/산출물/주의사항이 그 유형에 맞게 바뀐다.

### 이 스펙이 다루지 않는 것 (후속 SPEC)
- 문헌조사 보드 (후보기능 3)
- 가설-목표-방법 정렬 검사기의 고급 규칙 (후보기능 4) — 단, 캔버스 내 **기본 정합성 경고**는 본 스펙에 포함
- IRB/규제 허브, 예산/일정/리스크 패널, 연구계획서 초안 생성기

### 설계 원칙
- 기존 SPA 패턴 재사용: `data-tab` 글로벌 탭, `table-card`/`caution-card`/`modal-card`, `localStorage` 영속화, `render*` 함수 패턴.
- 신규 프레임워크/라이브러리 도입 금지. 순수 HTML/CSS/JS로 구현.
- 비개발자 사용성 우선: 한 화면에서 입력→경고→초안까지 끊기지 않게.
- 모든 입력은 로컬에만 저장(서버 없음). 개인정보/병원정보 하드코딩 금지.

## 기능 1: 연구 질문 캔버스

### 1.1 진입점 (UI 위치)
- 글로벌 네비게이션(`.nav-group` 좌측 사이드바)에 새 글로벌 탭 추가.
  - `data-tab="canvas"`, 아이콘 `🧭`, 라벨 `연구 질문 캔버스`.
  - 위치 권장: `종합 대시보드` 바로 아래(첫 진입 기능이므로 상단 노출).
- 탭 본문은 기존 `table-card` 컨테이너 스타일을 그대로 사용.

### 1.2 입력 필드 스키마

| # | 필드 ID | 라벨 | 입력 형태 | 필수 | 플레이스홀더/도움말 |
|---|---|---|---|---|---|
| 1 | `projectName` | 프로젝트명 | text(1줄) | 권장 | 예: 항암 후보물질 A 전임상 효능 평가 |
| 2 | `researchQuestion` | 연구 질문 | textarea | **필수** | 한 문장으로. 대상·비교·측정·성공판단이 보이게 |
| 3 | `hypothesis` | 가설 | textarea | **필수** | 왜 그런 결과가 나올 것이라 보는가(기전/방향) |
| 4 | `primaryOutcome` | 1차 결과변수 | text(1줄) | **필수** | 성공 여부를 판단할 단일 지표. 예: 12주 종양 크기 변화율 |
| 5 | `subjectModel` | 대상/모델 | text(1줄) | **필수** | 세포주·동물·환자군·데이터셋 |
| 6 | `comparator` | 비교군 | text(1줄) | **필수** | 무처치/vehicle/SoC/placebo/historical |
| 7 | `successCriteria` | 성공 기준 | textarea | **필수** | 어떤 수치/방향이면 성공이라 부를지 |
| 8 | `studyType` | 연구 유형 | select(아래 6택1) | 권장 | 기능 2 마법사와 동일 값. 미선택 가능 |
| 9 | `secondaryOutcomes` | 2차 결과변수 | text(다중, 선택) | 선택 | 쉼표 구분. 없으면 비움 |
| 10 | `expectedImpact` | 예상 임팩트 | textarea | 선택 | 성공 시 무엇을 주장할 수 있는가 |
| 11 | `mainRisk` | 주요 리스크 | textarea | 선택 | 가장 큰 실패 가능성 1~2개 |
| 12 | `goNoGo` | 다음 단계 Go/No-Go 기준 | textarea | 선택 | 어떤 결과면 다음 단계로 가는가 |

비고:
- 1~7번은 컨셉 노트의 핵심이므로 화면 상단 그룹.
- 8~12번은 “선택 입력(접기/펼치기)” 그룹으로 둬도 됨.
- `studyType` 값 목록은 기능 2의 `STUDY_TYPES` 키와 **반드시 동일**해야 함(중복 정의 금지, 단일 소스).

### 1.3 데이터 모델 & 영속화
- 단일 객체 `canvasState`로 관리.
- `localStorage` 키: `rd_canvas_state` (기존 `rd_lifecycle_checklist_states`, `rd_glossary_filters` 네이밍과 통일).
- 저장 시점: 입력 `change`/`blur` 시 디바운스 저장(기존 체크리스트 저장 방식과 동일한 즉시-영속 패턴).
- 스키마(JSON):

```json
{
  "projectName": "",
  "researchQuestion": "",
  "hypothesis": "",
  "primaryOutcome": "",
  "subjectModel": "",
  "comparator": "",
  "successCriteria": "",
  "studyType": "",
  "secondaryOutcomes": [],
  "expectedImpact": "",
  "mainRisk": "",
  "goNoGo": "",
  "updatedAt": "ISO8601"
}
```

### 1.4 검증 / 경고 규칙

경고는 **차단(block)이 아니라 안내(advisory)**다. 입력은 항상 가능하되, 경고 배너를 `caution-card` 스타일로 표시한다. 심각도 2단계: `warn`(노랑), `info`(파랑).

| 규칙 ID | 조건 | 심각도 | 메시지(사용자 노출 문구) | 근거 |
|---|---|---|---|---|
| C-01 | 필수 필드(2~7) 중 빈 항목 존재 | warn | "필수 항목이 비어 있습니다: {필드라벨 목록}. 비어 있으면 컨셉 노트가 불완전합니다." | 1장, 12장 |
| C-02 | `researchQuestion`에 비교 표현(대비/보다/대조/vs/비교)이 없음 | info | "연구 질문에 비교 기준이 보이지 않습니다. ‘표준 처리 대비’처럼 비교 대상을 넣으면 좋은 질문이 됩니다." | 1장(좋은 질문 기준) |
| C-03 | `comparator`가 비어 있는데 `researchQuestion`엔 비교 표현이 있음 | warn | "질문에는 비교가 있는데 비교군이 비어 있습니다. 비교군을 명시하세요." | 4장, 5-1장 |
| C-04 | `primaryOutcome`가 비어 있거나 측정 가능 표현이 약함(숫자/율/점수/시간/농도/생존 등 측정 키워드 없음) | info | "1차 결과변수가 ‘측정 가능한 지표’로 보이지 않습니다. 수치·비율·점수 등 측정 형태로 적으세요." | 4장, 6-1장 |
| C-05 | `successCriteria`에 방향/임계값 표현(이상/이하/%/감소/증가/p<)이 없음 | info | "성공 기준에 판단 임계값이 없습니다. ‘20% 이상 감소’처럼 합격선을 적으세요." | 4-3장, 10-5장 |
| C-06 | 가설-결과변수 정합성: `hypothesis`가 효능/억제/개선 방향인데 `primaryOutcome`가 안전성/독성 단어만 포함(또는 반대) | warn | "가설은 ‘효능’을 말하는데 1차 결과변수는 ‘안전성’만 측정합니다. 목표와 변수가 어긋날 수 있습니다." | 4장(정렬), 4-3장 |
| C-07 | `studyType` 미선택 | info | "연구 유형을 고르면 유형별 체크리스트와 주의사항을 함께 볼 수 있습니다." | 2장 |

키워드 사전(구현 참고, 한국어/영어 혼용 매칭, 대소문자 무시):
- 비교 표현: `대비, 보다, 대조, 비교, versus, vs, compared`
- 측정 키워드: `율, %, 퍼센트, 점수, score, 크기, 농도, 수치, 시간, 생존, 발현, IC50, ROC, 민감도, 특이도`
- 성공 임계값: `이상, 이하, 초과, 미만, %, 감소, 증가, p<, p <, 유의, 향상`
- 효능 방향: `효능, 억제, 감소, 개선, 효과, 치료, 반응`
- 안전성 방향: `안전성, 독성, 부작용, 이상반응, toxicity`

경고 영역 UI:
- 입력 그룹 하단에 `#canvasWarnings` 컨테이너.
- 경고 0건이면 “✅ 기본 정합성 점검 통과” 초록 배지.
- 경고는 규칙 ID 순 정렬, 심각도 아이콘(⚠️/ℹ️) 표시.
- 경고는 입력 변경 시 실시간 재평가(`evaluateCanvas()` 호출).

### 1.5 출력: 1페이지 컨셉 노트 초안 생성

- 버튼: `컨셉 노트 초안 만들기` (`#genConceptNote`).
- 동작:
  1. `C-01`(필수 빈칸) 경고가 있으면 생성은 허용하되, 노트 상단에 “⚠️ 미완성 초안” 배너와 빈 항목 목록을 포함.
  2. 결과를 읽기전용 `textarea`(또는 `modal-card`) `#conceptNoteOutput`에 렌더.
  3. `복사`(클립보드), `.txt 다운로드`, `.md 다운로드` 버튼 제공(D-4 확정). `.md`는 위 템플릿을 마크다운 헤더/리스트로 감싼 형식.
- 출력 템플릿 (가이드 10-1과 동일 골격):

```text
프로젝트명: {projectName}
배경과 문제: {expectedImpact 기반 1줄 또는 "(미작성)"}
핵심 연구 질문: {researchQuestion}
가설: {hypothesis}
대상/모델: {subjectModel}
비교군: {comparator}
핵심 방법: {studyType 표시명 + "(상세 방법은 후속 단계에서)"}
1차 결과변수: {primaryOutcome}
2차 결과변수: {secondaryOutcomes join ", "}
성공 기준: {successCriteria}
예상 임팩트: {expectedImpact}
주요 리스크: {mainRisk}
다음 단계 Go/No-Go 기준: {goNoGo}

(이 초안은 자동 생성본입니다. 통계·IRB·규제 담당자 검토 전 최종본이 아닙니다.)
```

### 1.6 사용자 흐름 (기능 1)

1. 사용자가 좌측 `연구 질문 캔버스` 탭 클릭 → 입력 폼 표시(이전 저장값 자동 복원).
2. 필드를 채우는 동안 하단 경고 영역이 실시간 갱신.
3. 경고를 보며 질문/가설/결과변수를 다듬음.
4. `컨셉 노트 초안 만들기` 클릭 → 노트 미리보기 표시.
5. 복사 또는 다운로드.
6. 새로고침해도 입력값 유지(localStorage).
7. (선택) `초기화` 버튼으로 캔버스 비우기 — 기존 체크리스트 초기화처럼 확인 후 실행.

## 기능 2: 연구 유형 분기형 마법사

### 2.1 목적
사용자가 연구 유형을 고르면 그 유형에 맞는 **체크리스트 / 핵심 산출물 / 대표 리스크·주의사항**을 보여준다. 가이드 2장 표를 데이터화한 것이다.

### 2.2 진입점 (D-1 확정: 옵션 A)
- **캔버스 탭 안 통합.** `studyType` select를 바꾸면 같은 화면 하단의 “유형별 가이드” 패널(`#studyTypePanel`)이 즉시 갱신/펼쳐짐.
- 별도 글로벌 탭은 만들지 않는다.
- 데이터 소스는 `STUDY_TYPES` 단일 상수.

### 2.3 데이터 모델: `STUDY_TYPES`

6개 유형. 키는 캔버스 `studyType` 값과 동일.

```js
const STUDY_TYPES = {
  basic:        { label: "기초연구",            coreQuestion: "기전이 존재하는가",
                  deliverables: ["실험 데이터", "그림", "논문 초안"],
                  risks: ["재현성 부족", "모델 부적합"],
                  checklist: [ /* 아래 2.4 */ ] },
  nonclinical:  { label: "비임상/전임상",        coreQuestion: "효능·독성·작용기전이 충분한가",
                  deliverables: ["효능/독성 보고서", "Go/No-Go 근거"],
                  risks: ["모델 외삽 실패", "독성 이슈"], checklist: [] },
  clinical:     { label: "임상연구",            coreQuestion: "사람에서 안전성·유효성이 있는가",
                  deliverables: ["프로토콜", "CRF", "통계계획", "임상결과"],
                  risks: ["대상자 모집", "규제 지연"], checklist: [] },
  observational:{ label: "관찰연구/실사용데이터", coreQuestion: "연관성·예후·패턴이 있는가",
                  deliverables: ["코호트 분석", "예측모형"],
                  risks: ["편향", "결측치", "혼란변수"], checklist: [] },
  diagnostic:   { label: "진단/바이오마커",      coreQuestion: "정확도와 임상적 유용성이 있는가",
                  deliverables: ["민감도/특이도", "ROC", "컷오프"],
                  risks: ["기준검사 부재", "과적합"], checklist: [] },
  industrial:   { label: "산업화 R&D",          coreQuestion: "제품·서비스로 전환 가능한가",
                  deliverables: ["개발계획", "IP", "규제전략", "사업계획"],
                  risks: ["시장 미스매치", "제조/품질 문제"], checklist: [] }

```

### 2.4 유형별 체크리스트(가이드 5-2장 기반)

- **기초/비임상 공통(basic, nonclinical)**:
  - [ ] 생물학적으로 타당한 모델을 선택했는가
  - [ ] 용량-반응 관계를 확보할 설계인가
  - [ ] 반복 실험과 독립 재현성을 확보했는가
  - [ ] 음성/양성 대조군을 준비했는가
  - (nonclinical 추가) [ ] 다음 단계 진입 Go/No-Go 근거를 정의했는가
- **clinical**:
  - [ ] Phase(1/2/3) 목적을 명확히 했는가
  - [ ] 참여기준·표본수·기간·대조군·투여법·평가시점을 사전 정의했는가
  - [ ] 안전성 모니터링/중단 기준이 있는가
  - [ ] IRB·동의서·규제(IND 등) 경로를 확인했는가
- **observational**:
  - [ ] 선택편향·정보편향·교란변수 통제 계획이 있는가
  - [ ] 결측치 처리 원칙을 사전 정의했는가
  - [ ] 분석 전 데이터 정의서를 확정했는가
- **diagnostic**:
  - [ ] 기준검사(reference standard)를 정의했는가
  - [ ] 검체 취급을 표준화했는가
  - [ ] 학습/검증 데이터를 분리했는가
  - [ ] 임상적 컷오프와 외부검증 계획이 있는가
- **industrial**:
  - [ ] 제품 분류(의약품/바이오/기기/진단)를 정했는가
  - [ ] 제조·품질(배치 일관성·안정성) 계획이 있는가
  - [ ] IP·규제전략·상업성을 함께 검토했는가
  - [ ] 과학적 성공 외 사업 전환 경로가 있는가

> 체크 상태도 `localStorage`에 유형별로 저장(`rd_studytype_checklist`), 기존 `checklistStates` 패턴 재사용. 유형 전환 시 해당 유형 상태를 로드.

### 2.5 분기 동작 규칙

| 규칙 ID | 트리거 | 동작 |
|---|---|---|
| W-01 | `studyType` 선택/변경 | 해당 유형의 coreQuestion·deliverables·risks·checklist 렌더 |
| W-02 | 유형 변경 | 이전 유형 체크 상태는 보존, 현재 유형 상태만 화면에 표시 |
| W-03 | 캔버스의 `comparator`가 비었고 유형이 `clinical`/`nonclinical` | 주의 배너: "효능 평가형 연구는 비교군 정의가 핵심입니다." |
| W-04 | 유형이 `clinical`/`observational`/`diagnostic`(사람 대상 가능) | info 배너: "사람 대상 가능성이 있어 IRB/규제 점검이 필요합니다(후속 IRB 허브 예정)." |
| W-05 | 유형이 `industrial` | info 배너: "과학적 성공뿐 아니라 제조·허가·상업성도 계획에 포함하세요." |

### 2.6 표시 구성(유형 선택 후 패널)
- 카드 1: `핵심 질문` (coreQuestion) — `detail-card` 스타일.
- 카드 2: `핵심 산출물` 칩 목록 (deliverables).
- 카드 3: `유형별 체크리스트` — 기존 `renderChecklist` 패턴 재사용.
- 카드 4: `대표 리스크 & 주의사항` — `caution-card` 스타일 + W-03~W-05 배너.

### 2.7 사용자 흐름 (기능 2)
1. 캔버스에서 `연구 유형` 선택(또는 별도 탭 진입).
2. 유형별 패널이 즉시 갱신(W-01).
3. 체크리스트를 점검하며 누락 사항 확인.
4. 주의 배너(W-03~05)로 유형 특이 리스크 인지.
5. 유형을 바꿔 비교해도 각 유형 체크 상태는 보존(W-02).

## 3. 기능 1·2 연동 요약
- `studyType`는 단일 소스(`STUDY_TYPES` 키). 캔버스 select와 마법사가 같은 값을 공유.
- 캔버스 입력(comparator 등)이 마법사 배너 조건(W-03)에 반영.
- 컨셉 노트의 “핵심 방법” 줄에 유형 라벨이 들어감.

## 4. 비기능 요구사항
- 오프라인 동작(로컬 파일 열기로 사용 가능).
- 기존 탭/단계 기능에 영향 없음(회귀 금지).
- 접근성: 입력에 `<label>`, 경고는 `aria-live="polite"`.
- 한국어 UI, 문구는 비개발자 친화적.

## 5. 수용 기준(Acceptance Criteria)
- AC-1: 캔버스 필드 입력값이 새로고침 후에도 유지된다.
- AC-2: 필수 항목 비면 C-01 경고가 보인다.
- AC-3: 가설=효능 / 결과변수=안전성 입력 시 C-06 경고가 뜬다.
- AC-4: `컨셉 노트 초안 만들기`가 10-1 템플릿 형식으로 출력하고 복사·다운로드된다.
- AC-5: 6개 유형 각각 선택 시 고유 체크리스트/산출물/리스크가 바뀐다.
- AC-6: 유형 전환 후 되돌아와도 각 유형 체크 상태가 보존된다.
- AC-7: 기존 단계 네비게이션·용어사전·계산기 기능이 그대로 동작(회귀 없음).

## 6. 구현 노트(Codex 참고)
- 신규 탭은 기존 `data-tab` 라우팅 함수에 케이스만 추가.
- 저장 키 네이밍: `rd_canvas_state`, `rd_studytype_checklist`.
- `STUDY_TYPES`는 `STAGES` 등 기존 데이터 상수 근처에 정의.
- 경고 평가는 순수 함수 `evaluateCanvas(state) -> [{id, severity, message}]`로 분리(테스트 용이).
- DOM 주입 시 사용자 입력은 `textContent`로 넣어 XSS 방지(혹시 향후 공유 기능 대비).

## 7. 테스트 제안(가벼운 수동 + 함수 단위)
- `evaluateCanvas`에 대한 케이스 테이블: 각 C-01~C-07 트리거 입력/비트리거 입력.
- 수동 체크: AC-1~AC-7을 브라우저에서 1회씩 확인(Antigravity 검증 단계).

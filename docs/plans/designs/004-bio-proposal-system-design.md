## 1. 아키텍처 의사결정 요약 (Design Decisions)

### 1.1 반자동화 템플릿 제어 (Semi-automated Workflow)
* **방침**: 모든 제안서 작성은 완전 자동 생성이 아닌, 에이전트가 문헌 정보 파싱, FTO 회피 분석, 표준 아웃라인 목차 매핑을 자동 수행하여 마크다운(.md) 초안을 도출하고 사용자가 편집·보완하는 반자동 방식으로 진행함.
* **이유**: 바이오 R&D 계획서 특유의 학술적 전문성과 주관기관 고유 데이터의 미세 조율이 필수적이기 때문임.

### 1.2 스타일 템플릿 기반 DOCX 매핑 (Output Format)
* **방침**: 작성 완료된 마크다운 결과물은 파이썬 스크립트(`python-docx` 활용)를 통해 사전에 스타일과 레이아웃이 설정된 DOCX 파일의 변수 영역에 자동 주입되어 변환됨.
* **이유**: HWP(한컴오피스)로 직접 텍스트 및 표 서식을 유지한 채 가져오기(Import)에 가장 유리하며 실무 가독성을 보장하기 위함임.

### 1.3 하이브리드 외부 데이터 API 연동 (Data Ingestion)
* **방침**: PubMed/PMC 및 USPTO는 인증키 없는 퍼블릭 API를 기본 작동하며, KIPRIS 및 ClinicalTrials 등은 `.env` 파일에 API 키가 기재되어 있으면 공식 조회를 수행하고, 없을 시 웹 스크래핑으로 자동 폴백 처리함.
* **이유**: 사용자의 불필요한 행정적 연동 절차를 단축하면서도 데이터 조회 수집력을 확보하기 위함임.

### 1.4 보안 옵트아웃 및 로컬 LLM 하이브리드 라우팅 (Security)
* **방침**: 클라우드 모델 사용 시 텔레메트리 전송을 강제 배제(`telemetry: false`)하고, 기밀 물질 정보가 집중 기술되는 2장(연구 내용) 작성 시에는 로컬 LLM(Ollama 등 로컬 구동 모델)으로 에이전트 라우팅을 실시간 전환할 수 있도록 환경을 개방함.
* **이유**: 핵심 후보물질의 특허 출원 전 유출 가능성을 원천 봉쇄하기 위함임.

### 1.5 다중 규격 RFP 파싱 (RFP Ingestion)
* **방침**: 사용자가 정부 공고문 PDF 또는 TXT 파일을 `source-papers/rfp.pdf` 또는 `source-papers/rfp.txt` 경로에 적재하면 파서 에이전트가 이를 읽어 필수 요구 요건을 리스트업함.
* **이유**: 부처별 공고 형식 다변화에 유연하게 대응하기 위함임.
